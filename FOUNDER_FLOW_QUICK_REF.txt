═══════════════════════════════════════════════════════════════════════════════
  2. CRITICAL CONSTRAINT: Plans Are Self-Modifying
═══════════════════════════════════════════════════════════════════════════════

PROBLEM:
  - Taskmaster READS plan file from disk
  - Taskmaster UPDATES checkboxes in-place (marks tasks done)
  - If user edits plan WHILE executing, corruption happens

IMPLICATION FOR FOUNDER FLOW:
  - Can't allow user to edit spec while execution is live
  - Need to separate "spec document" from "execution checkpoint"
  - Solution: Use .anyon/plans.{plan-name}.execution checkpoint file

═══════════════════════════════════════════════════════════════════════════════
  5. PROPOSED DATA FLOW (DETAILED)
═══════════════════════════════════════════════════════════════════════════════

PHASE 1: INTERVIEW & IDEATION
  User: "I want to build a dashboard"
    ↓
  Strategist: Asks clarifying questions
    ↓
  User: Answers (tech stack, features, etc)

PHASE 2: SPEC GENERATION (DRAFT)
  Strategist generates markdown spec → SAVED TO: .anyon/plans.draft/spec-{slug}.md
    ↓
  Metadata in YAML frontmatter:
    ├─ title: "Dashboard"
    ├─ status: "draft"           ← NEW
    ├─ version: 1                ← NEW
    ├─ chatId: 42
    ├─ createdAt: ISO
    └─ updatedAt: ISO

PHASE 2.5: SPEC REFINEMENT (USER EDITS)
  User reads spec in chat
    ↓
  User: "/edit-spec {section}" or "/edit-spec all"
    ↓
  Strategist: Regenerates section(s) → UPDATES draft file
    ↓
  User: Reads updated spec
    ↓
  Loop: Repeat until "let's go" or "/finalize-spec"

PHASE 3: FINALIZATION
  User: "/finalize-spec"
    ↓
  Analyst: Consults (gap analysis, guardrails)
    ↓
  Strategist: Prompts for critical decisions (if any)
    ↓
  User: Answers (or system applies defaults)
    ↓
  Strategist: Finalizes → COPIES draft → .anyon/plans/plan-{slug}.md
    ├─ status: "accepted"        ← CHANGED
    ├─ frozen: true              ← NEW
    ├─ version: 1 (locked)       ← NEW
    └─ checksummed              ← NEW (for integrity)

PHASE 4: START-WORK
  User: "/start-work dashboard"
    ↓
  start-work-hook.ts:
    ├─ Find .anyon/plans/plan-dashboard.md
    ├─ Check status != "draft" (GUARD)
    ├─ Create thesis.json:
    │   {
    │     active_plan: ".../plan-dashboard.md",
    │     status: "executing"
    │   }
    └─ Switch to Taskmaster agent

PHASE 5: EXECUTION
  Taskmaster:
    ├─ Read plan
    ├─ Create .anyon/plans/plan-dashboard.execution (checkpoint file)
    │   └─ Only THIS file gets checkbox updates
    ├─ Execute tasks
    ├─ Update .execution checkboxes
    └─ On complete: mark thesis.json status = "complete"

  │
  └─ thesis.json                  ← Tracks both draft and execution

═══════════════════════════════════════════════════════════════════════════════
  8. RISK SUMMARY
═══════════════════════════════════════════════════════════════════════════════

BIGGEST RISKS:
  1. Plan file corruption (user edits while executing)
     → Mitigated by: .execution checkpoint file
  
  2. User confusion (2 plan types)
     → Mitigated by: Clear UI labels (DRAFT / READY / EXECUTING)
  
  3. Draft proliferation (user creates too many)
     → Mitigated by: UI limit (show only 3-5 recent drafts)
  
  4. Lost work if user closes chat
     → Mitigated by: Auto-save on every edit

LOWEST RISK:
  - Analyst consultation already exists
  - Markdown format already extensible
  - IPC handlers already flexible

═══════════════════════════════════════════════════════════════════════════════
