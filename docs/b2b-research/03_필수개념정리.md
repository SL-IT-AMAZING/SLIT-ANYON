## 인프라 & 배포

### On-Premise vs Cloud vs Hybrid

**용어**: 온프레미스 vs 클라우드 vs 하이브리드 배포

**설명**:  
- **On-Premise**: 고객사 자체 서버실/IDC에 직접 설치. 고객이 모든 인프라 소유.
- **Cloud**: AWS, Azure, GCP 같은 외부 클라우드에서 운영.
- **Hybrid**: 민감한 데이터는 온프레미스, 나머지는 클라우드.

**고객 관심 이유**:  
금융, 공공기관, 대기업은 보안/규제 때문에 데이터를 외부로 못 보냅니다. 특히 개인정보, 기밀 정보는 반드시 내부에 두고 싶어 합니다.

**미팅 멘트**:  
- "저희 제품은 **온프레미스 배포를 완벽히 지원**합니다. 고객사 서버실에 직접 설치하시면 데이터가 외부로 나가지 않습니다."
- "현재 금융권 고객사에서는 하이브리드 구성으로 운영 중이시고, 민감 데이터는 온프레미스에서 처리하고 있습니다."
- "클라우드 배포도 지원하지만, 보안 요구사항 때문에 온프레미스를 선호하시는 기업이 많습니다."

---

### Docker / Container / Kubernetes

**용어**: 도커 / 컨테이너 / 쿠버네티스

**설명**:  
- **Docker**: 애플리케이션을 격리된 컨테이너로 패키징하는 도구.
- **Container**: 앱+라이브러리를 하나로 묶은 실행 환경. 어디서든 동일하게 실행됨.
- **Kubernetes (K8s)**: 많은 컨테이너를 자동으로 관리·배포·확장하는 오케스트레이션 시스템.

**고객 관심 이유**:  
설치가 복잡하면 IT팀이 싫어합니다. Docker로 제공하면 설치 한 줄로 끝나고, Kubernetes 지원하면 대규모 배포·장애 복구가 자동화됩니다.

**미팅 멘트**:  
- "저희는 **Docker 이미지로 제공**해서, 단 몇 분 만에 설치 가능합니다."
- "쿠버네티스 환경에서 운영 중이시면 **헬름 차트(Helm Chart)**도 제공해 드립니다."
- "컨테이너 기반이라 기존 인프라에 영향 없이 독립적으로 운영하실 수 있습니다."

---

### CI/CD

**용어**: CI/CD (Continuous Integration / Continuous Deployment, 지속적 통합/배포)

**설명**:  
코드 변경 → 자동 테스트 → 자동 배포까지 파이프라인으로 연결하는 개발 방식. 수동 배포 없이 안전하고 빠르게 업데이트 가능.

**고객 관심 이유**:  
기업은 "업데이트가 복잡하면 안 한다"는 문제가 있습니다. CI/CD가 있으면 버그 수정·기능 추가가 빠르고 안정적으로 반영됩니다.

**미팅 멘트**:  
- "저희는 **CI/CD 파이프라인을 갖춰서** 버그 수정이나 기능 추가 시 빠르게 배포해 드립니다."
- "고객사 내부 CI/CD 환경(Jenkins, GitLab CI 등)에 통합도 가능합니다."
- "업데이트 시 다운타임 없이 롤링 업데이트로 진행합니다."

---

### 망분리 (Air-Gap)

**용어**: 에어갭 / 망분리 환경

**설명**:  
외부 인터넷과 완전히 단절된 네트워크 환경. 보안이 극도로 중요한 곳(금융, 군, 공공기관)에서 사용.

**고객 관심 이유**:  
"우리 서버는 인터넷 안 됩니다"라는 고객이 많습니다. 이런 환경에서도 설치·운영 가능한지가 핵심입니다.

**미팅 멘트**:  
- "저희는 **망분리 환경에서도 설치 가능**합니다. 오프라인 설치 패키지를 USB로 전달해 드립니다."
- "외부 API 호출 없이 완전히 독립 실행 가능합니다."
- "라이선스 인증도 오프라인 방식으로 지원합니다."

---

### VPN, Reverse Proxy, Load Balancer

**용어**: VPN / 리버스 프록시 / 로드 밸런서

**설명**:  
- **VPN**: 외부에서 회사 내부망에 안전하게 접속하는 가상 사설망.
- **Reverse Proxy**: 외부 요청을 내부 서버로 전달하는 중간 게이트웨이 (보안·성능 향상).
- **Load Balancer**: 여러 서버에 트래픽을 분산시켜 부하 관리.

**고객 관심 이유**:  
대부분 기업은 이미 VPN·프록시·로드밸런서를 쓰고 있습니다. 우리 제품이 이 환경에서 잘 동작하는지 확인하고 싶어 합니다.

**미팅 멘트**:  
- "저희 제품은 **리버스 프록시 뒤에서도 문제없이 동작**합니다. Nginx, Apache 모두 테스트 완료했습니다."
- "HA 구성을 위해 로드 밸런서 뒤에 여러 인스턴스를 두셔도 됩니다."
- "VPN 환경에서 접속하는 사용자도 동일하게 사용 가능합니다."

---

### SSL/TLS, HTTPS

**용어**: SSL/TLS 인증서, HTTPS (암호화 통신)

**설명**:  
웹 통신을 암호화해서 중간에서 데이터를 가로채도 못 보게 하는 기술. HTTPS = HTTP + 암호화.

**고객 관심 이유**:  
기업 보안팀은 "내부에서도 HTTPS 써야 한다"고 요구합니다. 특히 개인정보 전송 시 필수.

**미팅 멘트**:  
- "저희는 **HTTPS를 기본으로 지원**하고, 고객사 인증서를 적용하실 수 있습니다."
- "Let's Encrypt 같은 무료 인증서도 자동 갱신 가능합니다."
- "내부망 전용 인증서(사설 인증서)도 설정 가능합니다."

---

### DNS, Subdomain Routing

**용어**: DNS, 서브도메인 라우팅

**설명**:  
- **DNS**: 도메인 이름(example.com)을 IP 주소로 바꿔주는 시스템.
- **Subdomain Routing**: app.company.com, admin.company.com처럼 서브도메인별로 다른 서비스 연결.

**고객 관심 이유**:  
기업은 "우리 도메인으로 쓰고 싶다" + "여러 팀이 서브도메인으로 나눠 쓰고 싶다"는 요구가 많습니다.

**미팅 멘트**:  
- "고객사 도메인으로 설정 가능합니다. 예를 들어 **app.회사명.com** 같은 형태로요."
- "팀별로 서브도메인 분리도 지원합니다. 예: hr.company.com, sales.company.com"
- "내부 DNS 서버와 연동해서 사용 가능합니다."

## 보안

### SSO (SAML, OIDC, OAuth2)

**용어**: SSO (Single Sign-On, 통합인증) - SAML, OIDC, OAuth2

**설명**:  
한 번 로그인하면 여러 시스템을 추가 로그인 없이 쓸 수 있는 기능.  
- **SAML**: 기업용 표준. 주로 대기업·공공기관.
- **OIDC/OAuth2**: 구글·MS 계정으로 로그인. 요즘 많이 씀.

**고객 관심 이유**:  
대기업은 이미 SSO 시스템(Okta, Azure AD, Google Workspace)을 쓰고 있습니다. 추가 계정 만들기 싫어하고, 통합 로그인 필수로 요구합니다.

**미팅 멘트**:  
- "저희는 **SAML 2.0과 OIDC를 모두 지원**합니다. Okta, Azure AD, Google Workspace와 바로 연동 가능합니다."
- "사용자는 회사 계정 하나로 저희 시스템까지 로그인 없이 접속하실 수 있습니다."
- "SSO 연동 시 IT팀 부담이 줄어들고, 퇴사자 계정 관리도 자동으로 됩니다."

---

### RBAC

**용어**: RBAC (Role-Based Access Control, 역할 기반 접근 제어)

**설명**:  
사용자를 역할(Role)로 그룹화하고, 역할마다 권한을 다르게 주는 방식.  
예: "관리자"는 모든 기능, "일반 사용자"는 읽기만.

**고객 관심 이유**:  
대기업은 팀·직급·부서별로 접근 권한을 세밀하게 나누고 싶어 합니다. "아무나 다 보면 안 돼요."

**미팅 멘트**:  
- "저희는 **RBAC 기반으로 권한 관리**가 가능합니다. 관리자, 편집자, 뷰어 등 역할별로 설정하실 수 있습니다."
- "부서별, 프로젝트별로 권한을 세밀하게 나눌 수 있습니다."
- "특정 데이터는 특정 역할만 접근 가능하도록 제한할 수 있습니다."

---

### 저장/전송 데이터 암호화

**용어**: Encryption at Rest / Encryption in Transit (저장 데이터 암호화 / 전송 데이터 암호화)

**설명**:  
- **저장 데이터 암호화**: DB·파일에 저장할 때 암호화. 하드디스크 도난 당해도 못 봄.
- **전송 데이터 암호화**: 네트워크로 보낼 때 암호화 (HTTPS, TLS).

**고객 관심 이유**:  
개인정보보호법·금융 규제 때문에 암호화는 필수입니다. 특히 "DB 백업 파일 유출되면?"에 대한 대응책.

**미팅 멘트**:  
- "저희는 **DB 암호화(AES-256)**를 기본 지원합니다. 백업 파일이 유출돼도 복호화 불가능합니다."
- "전송 구간은 TLS 1.2 이상으로 암호화됩니다."
- "암호화 키 관리는 고객사에서 직접 하실 수 있습니다 (KMS 연동 가능)."

---

### 개인정보보호법, GDPR

**용어**: 개인정보보호법 (한국), GDPR (유럽)

**설명**:  
개인정보를 다룰 때 지켜야 하는 법. 동의 없이 수집·이용 금지, 유출 시 벌금·처벌.  
GDPR은 유럽 기준이지만 글로벌 기업은 다 따릅니다.

**고객 관심 이유**:  
법무팀·보안팀이 제일 신경 쓰는 부분. "이거 쓰다가 벌금 나오면 누가 책임지나요?"

**미팅 멘트**:  
- "저희는 **개인정보보호법을 준수**하고 있으며, 데이터 최소 수집 원칙을 따릅니다."
- "사용자 동의 관리, 열람·삭제 요청 처리 기능을 제공합니다."
- "GDPR 요구사항도 충족하며, 글로벌 고객사에서도 사용 중입니다."

---

### Audit Logging

**용어**: 감사 로그 (Audit Log)

**설명**:  
"누가, 언제, 무엇을, 어떻게 했는지" 모든 행위를 기록하는 로그.  
예: "홍길동이 2026-02-12 15:30에 고객 정보 조회"

**고객 관심 이유**:  
보안 사고 발생 시 추적 필수. 규제·감사 대응용. "누가 데이터 빼갔는지" 확인 가능해야 함.

**미팅 멘트**:  
- "저희는 **모든 사용자 행위를 감사 로그로 기록**합니다. 보안 감사 시 제출 가능합니다."
- "로그는 수정·삭제 불가능하게 저장되며, 최소 1년 이상 보관 가능합니다."
- "특정 사용자의 행위를 검색·추적할 수 있습니다."

---

### Zero Trust

**용어**: 제로 트러스트 (Zero Trust Security)

**설명**:  
"내부망이라도 믿지 마라." 모든 접근 요청을 매번 인증·검증하는 보안 모델.  
전통적 방식은 "내부 = 안전"으로 봤지만, 요즘은 내부자 공격도 많아서 Zero Trust 선호.

**고객 관심 이유**:  
최신 보안 트렌드. CTO·보안팀이 "우리 제로 트러스트 도입 중인데 지원하나요?" 물어봅니다.

**미팅 멘트**:  
- "저희는 **제로 트러스트 아키텍처와 호환**됩니다. 모든 API 요청은 인증 토큰 필요합니다."
- "네트워크 위치와 무관하게 항상 신원 확인 후 접근 가능합니다."
- "VPN 없이도 안전하게 사용 가능하도록 설계됐습니다."

---

### SOC2, ISO 27001

**용어**: SOC2, ISO 27001 (보안 인증)

**설명**:  
- **SOC2**: 미국 기준 보안 감사 인증. SaaS 기업이 많이 받음.
- **ISO 27001**: 국제 정보보안 관리 표준. 한국 기업도 많이 받음.

**고객 관심 이유**:  
대기업·금융사는 "인증 있어야 계약한다"고 요구합니다. 인증 = 신뢰도.

**미팅 멘트**:  
- "저희는 현재 **ISO 27001 인증 준비 중**이며, 보안 정책은 국제 표준을 따릅니다."
- "SOC2 Type 2 인증도 올해 목표입니다."
- (인증 없으면) "아직 인증은 없지만, 보안 체크리스트는 ISO 27001 기준으로 운영 중입니다."

## 엔터프라이즈 소프트웨어

### SaaS vs PaaS vs IaaS

**용어**: SaaS / PaaS / IaaS

**설명**:  
- **SaaS**: 완성된 소프트웨어 서비스 (Gmail, Slack). 바로 사용.
- **PaaS**: 개발 플랫폼 제공 (Heroku, AWS Elastic Beanstalk). 코드만 올리면 됨.
- **IaaS**: 서버·네트워크 인프라만 제공 (AWS EC2, Azure VM). 직접 다 설치.

**고객 관심 이유**:  
우리 제품이 어느 형태인지 이해해야 계약·구매 프로세스가 결정됩니다.

**미팅 멘트**:  
- "저희는 **SaaS 형태로 제공**하지만, 온프레미스 배포 시에는 PaaS처럼 사용하실 수 있습니다."
- "고객사가 IaaS(AWS, Azure)를 쓰고 계시면 저희가 설치·운영을 도와드립니다."
- "클라우드든 온프레미스든 동일한 기능을 제공합니다."

---

### Multi-Tenancy

**용어**: 멀티테넌시 (Multi-Tenancy, 다중 임차)

**설명**:  
하나의 시스템에서 여러 고객(Tenant)이 데이터·설정을 완전히 분리해서 사용하는 구조.  
예: A회사와 B회사가 같은 서버 쓰지만 서로의 데이터는 절대 안 보임.

**고객 관심 이유**:  
기업은 "다른 회사 데이터가 섞이면 안 돼요"라고 걱정합니다. 멀티테넌시 설계가 잘 됐는지 확인.

**미팅 멘트**:  
- "저희는 **멀티테넌시 구조**로 설계돼서, 각 고객사 데이터는 물리적으로 분리됩니다."
- "설정·사용자·데이터 모두 테넌트별로 완전히 격리돼 있습니다."
- "원하시면 DB도 테넌트별로 별도 인스턴스 사용 가능합니다."

---

### SLA

**용어**: SLA (Service Level Agreement, 서비스 수준 계약)

**설명**:  
"우리 서비스는 99.9% 가동률을 보장합니다" 같은 약속을 문서화한 것.  
목표 미달 시 환불·보상 조건도 포함.

**고객 관심 이유**:  
대기업은 "시스템 다운되면 우리 사업 멈춰요"라서 가동률 보장 계약을 원합니다.

**미팅 멘트**:  
- "저희는 **99.9% 가동률 SLA**를 제공합니다. 월 최대 43분 다운타임까지 허용."
- "SLA 미달 시 크레딧 환급 정책이 있습니다."
- (온프레미스) "온프레미스는 고객사 인프라에 의존하지만, 소프트웨어 자체는 고가용성(HA) 구성 가능합니다."

---

### API Gateway

**용어**: API 게이트웨이

**설명**:  
모든 API 요청의 진입점. 인증·로깅·속도 제한·라우팅을 중앙에서 관리.  
클라이언트는 게이트웨이 하나만 보면 되고, 내부는 여러 서비스로 분산 가능.

**고객 관심 이유**:  
기업은 API로 다른 시스템과 연동하고 싶어 합니다. 게이트웨이 있으면 관리·보안이 편함.

**미팅 멘트**:  
- "저희는 **RESTful API를 제공**하며, API 게이트웨이를 통해 안전하게 호출하실 수 있습니다."
- "API 키 인증, Rate Limiting, 로깅 모두 지원합니다."
- "고객사 API Gateway(Kong, AWS API Gateway)와 통합 가능합니다."

---

### Microservices vs Monolith

**용어**: 마이크로서비스 vs 모놀리식 (Monolith)

**설명**:  
- **Monolith**: 모든 기능이 하나의 큰 애플리케이션. 배포 간단하지만 수정 어려움.
- **Microservices**: 기능별로 작은 서비스들로 분리. 독립 배포·확장 가능하지만 복잡함.

**고객 관심 이유**:  
대기업은 "일부 기능만 업데이트하고 싶다" "특정 서비스만 확장하고 싶다"는 요구가 있습니다.

**미팅 멘트**:  
- "저희는 **마이크로서비스 아키텍처**로 설계돼서, 필요한 모듈만 선택적으로 배포하실 수 있습니다."
- "Auth, DB, AI 엔진 등이 독립 서비스라서, 부분 장애 시에도 전체가 멈추지 않습니다."
- (모놀리식이면) "저희는 모놀리식이지만, 컨테이너로 패키징돼서 배포·확장이 간단합니다."

---

### Webhook

**용어**: 웹훅 (Webhook)

**설명**:  
"이벤트 발생 시 자동으로 외부 URL로 알림 보내기."  
예: 슬랙에 메시지 보내기, 외부 시스템에 데이터 동기화.

**고객 관심 이유**:  
기업은 "우리 시스템과 연동하고 싶다"는 요구가 많습니다. Webhook이 제일 쉬운 연동 방법.

**미팅 멘트**:  
- "저희는 **Webhook을 지원**해서, 이벤트 발생 시 고객사 시스템으로 실시간 알림 보낼 수 있습니다."
- "예를 들어 신규 데이터 생성 시 슬랙·이메일·외부 API 호출 등 가능합니다."
- "커스텀 Webhook도 설정 가능합니다."

---

### ETL

**용어**: ETL (Extract, Transform, Load)

**설명**:  
- **Extract**: 여러 소스에서 데이터 추출.
- **Transform**: 데이터 정제·가공·변환.
- **Load**: 목적지(DB, Data Warehouse)에 적재.

**고객 관심 이유**:  
기업은 기존 시스템(ERP, CRM)의 데이터를 우리 시스템으로 가져오고 싶어 합니다.

**미팅 멘트**:  
- "저희는 **CSV, Excel, API를 통한 데이터 임포트**를 지원합니다. ETL 파이프라인 구성 가능합니다."
- "고객사 DB에서 주기적으로 데이터를 동기화하는 스케줄러도 제공합니다."
- "데이터 변환 로직은 고객사 요구에 맞춰 커스터마이징 가능합니다."

## AI/LLM

### RAG

**용어**: RAG (Retrieval-Augmented Generation, 검색 증강 생성)

**설명**:  
LLM이 답변할 때, 외부 문서·DB에서 관련 정보를 검색해서 참고하는 방식.  
LLM의 지식 제한을 극복. 회사 내부 문서·매뉴얼 기반 답변 가능.

**고객 관심 이유**:  
"우리 회사 데이터로 AI를 학습시키고 싶다" → Fine-Tuning은 비싸고 어려움. RAG가 현실적 대안.

**미팅 멘트**:  
- "저희는 **RAG 방식으로 고객사 문서를 AI에 학습**시킬 수 있습니다. 파인튜닝 없이도 가능합니다."
- "회사 매뉴얼, 계약서, FAQ 등을 업로드하면 AI가 그 내용 기반으로 답변합니다."
- "실시간으로 문서 업데이트 시 AI가 즉시 반영합니다."

---

### Fine-Tuning vs Prompt Engineering

**용어**: 파인튜닝 vs 프롬프트 엔지니어링

**설명**:  
- **Fine-Tuning**: 모델을 직접 재학습시켜 특정 스타일·지식 반영. 비싸고 시간 오래 걸림.
- **Prompt Engineering**: 프롬프트(질문 방식)를 잘 작성해서 원하는 답변 유도. 빠르고 저렴.

**고객 관심 이유**:  
"AI를 우리 회사 스타일로 바꾸고 싶다" → 대부분은 프롬프트로 해결됨. 파인튜닝은 극소수.

**미팅 멘트**:  
- "대부분 요구사항은 **프롬프트 엔지니어링으로 해결** 가능합니다. 예를 들어 '~한 톤으로 답변해줘' 같은 지시."
- "파인튜닝은 특수한 전문 용어·도메인 지식이 필요한 경우만 고려합니다. 비용·시간이 많이 듭니다."
- "저희는 프롬프트 템플릿을 제공해서 고객사가 쉽게 커스터마이징 가능합니다."

---

### LLM, Token, Context Window

**용어**: LLM, 토큰, 컨텍스트 윈도우

**설명**:  
- **LLM**: Large Language Model. GPT, Claude 같은 대형 언어 모델.
- **Token**: LLM이 처리하는 최소 단위. 한글 1글자 ≈ 2~3 토큰. 비용·속도에 영향.
- **Context Window**: LLM이 한 번에 볼 수 있는 최대 토큰 수. 예: GPT-4는 128K 토큰.

**고객 관심 이유**:  
"긴 문서를 AI가 다 읽을 수 있나요?" "비용이 얼마나 드나요?" → Context Window와 Token 개념 이해 필요.

**미팅 멘트**:  
- "저희가 사용하는 LLM은 **200K 토큰의 컨텍스트 윈도우**를 지원해서, 긴 문서도 한 번에 처리 가능합니다."
- "토큰 기반 과금이라서, 사용한 만큼만 비용이 발생합니다."
- "고객사 문서가 너무 길면 RAG로 필요한 부분만 검색해서 토큰을 절약합니다."

---

### Embedding, Vector Database

**용어**: 임베딩, 벡터 데이터베이스

**설명**:  
- **Embedding**: 텍스트를 숫자 벡터로 변환. 의미적으로 비슷한 문장은 비슷한 벡터.
- **Vector DB**: 임베딩 벡터를 저장하고, 유사도 검색을 빠르게 하는 DB (Pinecone, Weaviate, ChromaDB).

**고객 관심 이유**:  
RAG 구현 시 필수. "어떻게 관련 문서를 빠르게 찾냐?" → Vector DB 사용.

**미팅 멘트**:  
- "저희는 **벡터 데이터베이스(ChromaDB)**를 사용해서, 고객사 문서를 의미 기반으로 검색합니다."
- "키워드 매칭이 아니라 의미적 유사도로 찾아서, 표현이 달라도 관련 문서를 찾습니다."
- "온프레미스 배포 시에도 벡터 DB를 함께 설치해 드립니다."

---

### On-Premise LLM (Ollama, vLLM)

**용어**: 온프레미스 LLM - Ollama, vLLM

**설명**:  
OpenAI API 대신, 고객사 서버에 직접 LLM을 설치·운영하는 방식.  
- **Ollama**: 로컬 LLM 실행 도구. Llama, Mistral 등 오픈소스 모델 지원.
- **vLLM**: 빠른 추론을 위한 LLM 서빙 엔진.

**고객 관심 이유**:  
금융·공공은 "데이터를 외부 API(OpenAI)로 보낼 수 없다"는 규제. 온프레미스 LLM 필수.

**미팅 멘트**:  
- "저희는 **온프레미스 LLM을 지원**합니다. Ollama로 Llama 3 모델을 고객사 서버에 설치해 드립니다."
- "외부 API 호출 없이 완전히 내부에서만 AI가 동작합니다."
- "성능은 GPU 사양에 따라 다르지만, A100 기준 실시간 응답 가능합니다."

---

### AI Agent

**용어**: AI 에이전트

**설명**:  
단순 질의응답이 아니라, 목표를 주면 스스로 계획 세우고 여러 도구 사용하며 작업 수행하는 AI.  
예: "경쟁사 분석 보고서 작성해줘" → 웹 검색 + 데이터 분석 + 문서 작성.

**고객 관심 이유**:  
"AI가 단순 챗봇이 아니라 실제 업무를 대신 해줄 수 있나요?" → Agent 개념 중요.

**미팅 멘트**:  
- "저희 AI는 **에이전트 방식**으로 동작합니다. 단순 답변이 아니라 여러 단계 작업을 자동 수행합니다."
- "예를 들어 '데이터 분석해서 보고서 만들어줘'라고 하면, 데이터 조회 → 분석 → 차트 생성 → 문서화까지 자동입니다."
- "고객사 툴(API, DB)과 연동해서 에이전트가 직접 실행 가능합니다."

---

### MCP

**용어**: MCP (Model Context Protocol)

**설명**:  
Anthropic이 만든 표준 프로토콜. LLM이 외부 도구·데이터·API와 안전하게 연동하기 위한 규약.  
에이전트가 여러 시스템과 통신할 때 사용.

**고객 관심 이유**:  
"AI가 우리 시스템과 어떻게 연동하나요?" → MCP 같은 표준화된 방식이 있으면 신뢰도 UP.

**미팅 멘트**:  
- "저희는 **MCP(Model Context Protocol)**를 지원해서, AI가 고객사 시스템과 표준화된 방식으로 통신합니다."
- "보안과 권한 관리가 프로토콜 레벨에서 보장됩니다."
- "향후 다른 AI 시스템과 호환도 가능합니다."

## 로우코드/노코드

### Visual Builder vs Code

**용어**: 비주얼 빌더 vs 코드 작성

**설명**:  
- **Visual Builder**: 드래그앤드롭으로 UI·로직 구성. 코딩 몰라도 됨.
- **Code**: 직접 코드 작성. 자유도 높지만 개발자 필요.

**고객 관심 이유**:  
"우리 팀에 개발자 없는데 쓸 수 있나요?" vs "개발자가 코드로 세밀하게 컨트롤하고 싶어요."

**미팅 멘트**:  
- "저희는 **비주얼 빌더와 코드 에디터 모두 제공**합니다. 비개발자는 드래그앤드롭, 개발자는 코드 직접 작성 가능."
- "80%는 비주얼로 빠르게 만들고, 나머지 20%는 코드로 커스터마이징하는 하이브리드 방식이 가장 인기 있습니다."
- "코드 없이도 복잡한 워크플로우 구성 가능합니다."

---

### WYSIWYG

**용어**: WYSIWYG (What You See Is What You Get, 위지윅)

**설명**:  
"보이는 대로 결과물이 나온다." 실시간으로 결과를 보면서 디자인·편집하는 방식.  
예: 워드프로세서, 웹 페이지 빌더.

**고객 관심 이유**:  
비개발자가 쓰려면 WYSIWYG 필수. "Preview 보면서 수정하고 싶다."

**미팅 멘트**:  
- "저희는 **WYSIWYG 에디터**를 제공해서, 실시간으로 결과를 보면서 편집하실 수 있습니다."
- "버튼 위치, 색상, 레이아웃을 바로바로 확인하며 수정 가능합니다."
- "코드 몰라도 직관적으로 사용 가능합니다."

---

### Workflow Automation

**용어**: 워크플로우 자동화

**설명**:  
반복 작업을 자동화. "A 발생 → B 실행 → C 알림" 같은 흐름을 시각적으로 설계.  
예: Zapier, n8n, Make.

**고객 관심 이유**:  
기업은 "수작업 줄이고 싶다" "승인 프로세스 자동화하고 싶다" 같은 요구 많음.

**미팅 멘트**:  
- "저희는 **워크플로우 자동화 기능**을 제공합니다. 예를 들어 '신청서 제출 → 팀장 승인 → 이메일 발송' 같은 흐름을 자동화할 수 있습니다."
- "조건 분기, 반복, 지연 실행 등 복잡한 로직도 비주얼 빌더로 구성 가능합니다."
- "외부 API·웹훅과 연동해서 다른 시스템과도 자동화 가능합니다."

---

### Internal Tools

**용어**: 인터널 툴 (Internal Tools, 사내 도구)

**설명**:  
회사 내부에서만 쓰는 관리 도구. 예: 재고 관리, 고객 데이터 조회, 승인 시스템.  
공개 서비스 아니라서 보안·UX 기준이 다름.

**고객 관심 이유**:  
"외부 용도 아니라 직원들이 쓸 도구를 빠르게 만들고 싶다" → 로우코드가 딱 맞음.

**미팅 멘트**:  
- "저희는 **사내 도구 개발에 특화**돼 있습니다. 관리자 페이지, 데이터 조회 툴 등을 빠르게 만드실 수 있습니다."
- "외부 공개용이 아니라서 복잡한 인증·디자인 신경 안 쓰셔도 됩니다."
- "기존 DB·API와 연결해서 바로 사용 가능합니다."

### Citizen Developer

**용어**: 시티즌 개발자

**설명**:

## 혼동하면 안 되는 개념 쌍

### 1. Docker vs Kubernetes

- **Docker**: 컨테이너 하나를 만들고 실행하는 도구.
- **Kubernetes**: 수백 개 컨테이너를 자동으로 배포·관리·확장하는 오케스트레이션 플랫폼.

**비유**: Docker = 택배 상자 포장 / Kubernetes = 물류 센터 전체 관리

**미팅에서**:  
"Docker로 패키징해서 제공하고, Kubernetes 환경에서 대규모 배포·HA 구성 가능합니다."

---

### 2. SAML vs OAuth2

- **SAML**: 엔터프라이즈용 SSO 표준. 주로 회사 내부 시스템 통합.
- **OAuth2**: 권한 위임 프로토콜. "구글 계정으로 로그인" 같은 거.

**비유**: SAML = 회사 출입증 / OAuth2 = 외부 서비스 연동 키

**미팅에서**:  
"회사 SSO는 SAML로 연동하고, 외부 서비스(구글, MS)는 OAuth2로 지원합니다."

---

### 3. Fine-Tuning vs RAG

- **Fine-Tuning**: 모델 자체를 재학습. 비싸고 시간 오래 걸림.
- **RAG**: 외부 문서 검색해서 참고. 빠르고 저렴.

**비유**: Fine-Tuning = 대학원 가서 전공 공부 / RAG = 구글 검색하면서 답변

**미팅에서**:  
"대부분 고객사는 RAG로 충분하고, 극히 특수한 도메인만 Fine-Tuning 고려합니다."

---

### 4. API vs Webhook

- **API**: 내가 필요할 때 외부 서버에 요청.
- **Webhook**: 외부에서 이벤트 발생 시 내게 자동으로 알림.

**비유**: API = 전화 걸기 / Webhook = 전화 받기

**미팅에서**:  
"API는 고객사가 저희 데이터를 조회하고, Webhook은 저희가 이벤트 발생 시 고객사로 푸시합니다."

---

### 5. On-Premise vs Private Cloud

- **On-Premise**: 고객사 서버실에 물리 서버 설치.
- **Private Cloud**: AWS/Azure에서 고객 전용 격리 환경 (VPC).

**비유**: On-Premise = 자가용 / Private Cloud = 전세 택시

**미팅에서**:  
"완전 망분리면 On-Premise, 클라우드 쓰되 격리 원하시면 Private Cloud 제안드립니다."

---

### 6. Load Balancer vs Reverse Proxy

- **Load Balancer**: 여러 서버에 트래픽 분산.
- **Reverse Proxy**: 외부 요청을 내부 서버로 중계. 보안·캐싱 기능.

**비유**: Load Balancer = 은행 창구 배정 / Reverse Proxy = 은행 입구 안내 데스크

**미팅에서**:  
"Reverse Proxy(Nginx) 뒤에 저희 앱 두시고, Load Balancer로 여러 인스턴스 분산 가능합니다."

---

### 7. SaaS vs PaaS vs IaaS

- **SaaS**: 완성된 소프트웨어 (Gmail, Slack).
- **PaaS**: 개발 플랫폼 (Heroku, Google App Engine).
- **IaaS**: 서버·스토리지만 제공 (AWS EC2, Azure VM).

**비유**: SaaS = 완성 요리 / PaaS = 조리 도구 제공 / IaaS = 부엌만 빌려줌

**미팅에서**:  
"저희는 SaaS로 제공하지만, PaaS처럼 커스터마이징도 가능하고, IaaS에 직접 설치도 지원합니다."

---

### 8. Embedding vs Token

- **Token**: LLM이 처리하는 최소 단위 (글자 조각).
- **Embedding**: 텍스트를 의미 담은 숫자 벡터로 변환.

**비유**: Token = 단어 / Embedding = 단어의 의미를 숫자로 표현

**미팅에서**:  
"Token은 비용·속도와 관련되고, Embedding은 검색·유사도 계산에 씁니다."

---

### 9. Microservices vs API Gateway

- **Microservices**: 기능별로 서비스 분리하는 아키텍처 스타일.
- **API Gateway**: 모든 API 요청의 진입점. 인증·로깅 담당.

**비유**: Microservices = 여러 매장 / API Gateway = 백화점 안내 데스크

**미팅에서**:  
"저희는 마이크로서비스로 구성돼 있고, API Gateway를 통해 안전하게 호출하십니다."

---

### 10. SSO vs MFA

- **SSO**: 한 번 로그인으로 여러 시스템 접속.
- **MFA**: 비밀번호 + 추가 인증(OTP, 지문) = 다중 인증.

**비유**: SSO = 마스터 키 / MFA = 이중 잠금장치

**미팅에서**:  
"SSO로 편의성 높이고, MFA로 보안 강화하는 게 베스트 프랙티스입니다."

---

### 11. Monolith vs Distributed System

- **Monolith**: 모든 기능이 하나의 애플리케이션.
- **Distributed System**: 기능이 여러 서버·서비스에 분산.

**비유**: Monolith = 단독 주택 / Distributed = 아파트 단지

**미팅에서**:  
"모놀리식이 배포는 간단하지만, 분산 시스템이 확장성·장애 격리에 유리합니다."

---

### 12. Audit Log vs Application Log

- **Audit Log**: 보안·컴플라이언스용. "누가 무엇을 했는지" 기록.
- **Application Log**: 디버깅용. 에러·성능 정보.

**비유**: Audit Log = CCTV 녹화 / Application Log = 개발자 메모

**미팅에서**:  
"Audit Log는 삭제 불가·변조 방지로 보관하고, Application Log는 디버깅용입니다."

---

### 13. RAG vs Knowledge Base

- **RAG**: AI가 외부 문서 검색해서 답변 생성하는 기술.
- **Knowledge Base**: 구조화된 FAQ·문서 저장소.

**비유**: RAG = 사서가 책 찾아주는 도서관 / Knowledge Base = 책장

**미팅에서**:  
"Knowledge Base에 문서 저장하면, RAG가 그걸 검색해서 AI 답변에 활용합니다."

---

### 14. PoC vs Pilot vs Production

- **PoC**: 기술 검증. 소규모, 단기간.
- **Pilot**: 실제 사용자 일부 대상 테스트.
- **Production**: 전사 확대 배포.

**비유**: PoC = 시식 / Pilot = 일부 매장 출시 / Production = 전국 출시

**미팅에서**:  
"먼저 2주 PoC로 기술 검증 후, 한 팀 대상 Pilot, 그 다음 전사 Production 제안드립니다."

---

### 15. Horizontal Scaling vs Vertical Scaling

- **Horizontal**: 서버 대수 늘리기 (Scale Out).
- **Vertical**: 서버 성능 올리기 (Scale Up, CPU·RAM 증설).

**비유**: Horizontal = 점원 늘리기 / Vertical = 점원 능력치 올리기

**미팅에서**:  
"트래픽 증가 시 Horizontal Scaling으로 서버 추가하면 됩니다. Kubernetes면 자동 확장도 가능합니다."

## 추가: 실전 Q&A 대비

### Q1: "우리 회사는 AWS 쓰는데, 온프레미스만 되나요?"

**A**: "아닙니다. AWS, Azure, GCP 모두 지원합니다. 클라우드에 설치하셔도 데이터는 고객사 VPC 안에만 있어서 안전합니다. 원하시면 SaaS 형태로 저희가 호스팅해 드릴 수도 있습니다."

---

### Q2: "AI가 코드 짜주면 품질이 안 좋지 않나요?"

**A**: "AI가 초안을 빠르게 만들고, 개발자가 리뷰·수정하는 방식입니다. 오히려 반복 작업은 AI가 더 정확하고, 개발자는 핵심 로직에 집중할 수 있습니다. 코드 리뷰·테스트 기능도 제공합니다."

---

### Q3: "기존 시스템(ERP, 그룹웨어)과 연동되나요?"

**A**: "네. RESTful API, SOAP, DB 직접 연결 모두 지원합니다. 레거시 시스템 연동 경험 많아서, 기술팀과 협의해서 연동 방안 제안드리겠습니다."

---

### Q4: "사용자 100명, 1000명일 때 성능 차이는?"

**A**: "Horizontal Scaling으로 서버 추가하면 선형적으로 성능 늘어납니다. 1000명 규모도 레퍼런스 있고, 부하 테스트 결과 공유 가능합니다. PoC 때 고객사 환경에서 직접 측정해 보시는 게 정확합니다."

---

### Q5: "오픈소스 쓰면 무료 아닌가요?"

**A**: "초기는 무료지만, 엔터프라이즈 기능(SSO, RBAC, Audit Log)은 유료입니다. 또 설치·운영·커스터마이징에 내부 인력 투입 비용 생각하면, 상용 제품이 TCO가 낮은 경우 많습니다. ROI 시뮬레이션 도와드릴게요."

---

### Q6: "AI 학습에 우리 데이터 쓰이나요?"

**A**: "절대 안 됩니다. 온프레미스 LLM 쓰시면 데이터가 외부로 나가지 않고, 클라우드 API 쓰셔도 학습에는 사용 안 됩니다. 계약서에 명시 가능합니다."

---

### Q7: "업데이트는 어떻게 하나요?"

**A**: "Docker 이미지 교체 방식이라 다운타임 없이 롤링 업데이트 가능합니다. 자동 업데이트 기능도 있고, 수동으로 하실 수도 있습니다. 중요 업데이트는 사전 공지해 드립니다."

---

### Q8: "장애 나면 어떻게 대응하나요?"

**A**: "24/7 기술 지원 가능하고, SLA 보장합니다. 모니터링 대시보드 제공해서 실시간 상태 확인 가능하고, Critical 이슈는 1시간 내 대응합니다."

---

### Q9: "커스터마이징 가능한가요?"

**A**: "네. UI·워크플로우는 고객사가 직접 커스터마이징 가능하고, 코드 레벨 수정 필요하면 저희가 개발 지원해 드립니다. 별도 견적 후 진행됩니다."

---

### Q10: "PoC 기간·비용은?"

**A**: "보통 2주~1개월이고, 소규모는 무상 PoC 가능합니다. 대규모·커스터마이징 포함 시에는 비용 발생할 수 있습니다. 구체적인 범위 말씀해 주시면 견적 드리겠습니다."
