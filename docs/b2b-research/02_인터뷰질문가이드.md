# B2B 고객 발견 인터뷰 가이드
## AI 기반 사내 앱 빌더 (온프레미스)

> **목적**: 판매가 아닌 학습. 고객의 진짜 문제를 깊이 이해하기.  
> **대상**: IT 팀장, CTO, 개발팀 리더, 사내 시스템 담당자  
> **소요시간**: 약 60분

---

## 1. 인터뷰 전 준비사항

### 회사 리서치 체크리스트

**사전 조사 필수 항목** (30분 투자):
- [ ] 회사 업종/규모/직원 수 확인 (홈페이지, 사람인, LinkedIn)
- [ ] IR 자료/뉴스 검색 (최근 1년 내 디지털 전환 언급)
- [ ] 채용 공고 확인 (개발자 채용 여부, 기술 스택)
- [ ] 경쟁사 솔루션 사용 여부 (RetoolKR, 아웃코드, Bubble 등)
- [ ] 최근 IT 투자 뉴스 (ERP 도입, 클라우드 전환 등)
- [ ] 인터뷰이 LinkedIn 프로필 (경력, 관심사)

**왜 중요한가?**  
→ 사전 조사 없이 가면 "우리 회사도 모르고 왔네?"라는 인상. 신뢰 형성 실패.

---

### 준비할 자료

**반드시 챙겨갈 것**:
1. **명함** (최소 3장)
2. **노트북 + 충전기** (데모 요청 시 대비)
3. **노트 + 펜** (녹음 외 메모용)
4. **녹음기 or 스마트폰** (사전 허락 필수)
5. **1장짜리 제품 소개서** (요청 시만 제공, PDF로도 준비)
6. **질문 가이드 인쇄본** (A4 2장 요약본)

**선택 사항**:
- 유사 사례 스크린샷 (허락받은 레퍼런스만)
- 간단한 데모 영상 (1분 이내)

---

### 이메일 약속잡기 템플릿

**제목**: [학습 목적] 사내 개발 도구 현황 인터뷰 요청 - 30분

```
안녕하세요, [이름]님.

저는 기업 내부 개발팀의 생산성 향상을 돕는 AI 기반 앱 빌더를 
개발하고 있는 [본인 이름]입니다.

현재 제품 개발 전 고객 발견(Customer Discovery) 단계로,
실제 현장에서 겪는 어려움을 듣고 배우는 시간을 갖고자 합니다.

**판매나 영업 목적이 아니며**, 순수하게 배움의 기회를 얻고자 합니다.

인터뷰 내용:
- 현재 사내 도구 개발 프로세스와 어려움
- AI 도구 활용 현황 및 보안 정책
- 온프레미스 환경에서의 개발/배포 경험

소요시간: 약 30-40분
장소: [회사명] 사무실 또는 편하신 장소
일정: [3개 후보 날짜/시간 제시]

예) 
- 12월 15일(월) 오후 2시
- 12월 17일(수) 오전 11시
- 12월 19일(금) 오후 4시

귀한 시간 내주시면 감사하겠습니다.
인터뷰 참여에 대한 감사의 표시로 소정의 답례품을 준비하겠습니다.

[본인 이름]
[연락처]
[회사명/소속]
```

**팁**: 
- "판매 아님"을 명확히 강조
- 3개 시간대 제시 (선택권 제공)
- 답례품 언급 (스타벅스 기프티콘 1-2만원)

## 2. 오프닝 스크립트 (5분)

### 인사 + 명함 교환

**스크립트**:
```
안녕하세요, [이름]님. 만나 뵙게 되어 반갑습니다.
바쁘신데 시간 내주셔서 정말 감사합니다.

[명함 건네며] 저는 [본인 소개 - 1문장]입니다.
명함 받아도 될까요?

[자리 앉으며] 여기까지 오시는 데 불편하지는 않으셨죠?
```

**몸짓**:
- 눈 맞춤 + 미소
- 명함은 두 손으로
- 받은 명함은 테이블에 정중히 놓기

---

### 목적 설명 ("판매 아닌 학습")

**스크립트**:
```
오늘 인터뷰의 목적을 먼저 말씀드리고 싶습니다.

저희는 지금 제품을 '만들기 전' 단계에 있습니다.
그래서 오늘은 판매나 영업이 아니라, 
진짜 현장에서 어떤 어려움이 있는지 배우러 왔습니다.

솔직하게 말씀해 주시면 정말 큰 도움이 됩니다.
"이건 안 좋다", "이건 필요 없다" 같은 부정적인 의견도
오히려 더 귀합니다.

혹시 궁금하신 부분 있으세요?
```

**왜 중요한가?**  
→ 심리적 방어막 제거. "세일즈 당하겠구나"라는 경계심 해소.

---

### 녹음 허락

**스크립트**:
```
그리고 한 가지 더 여쭤보고 싶은데요,
대화 내용을 녹음해도 괜찮으실까요?

메모만으로는 놓치는 부분이 많아서요.
녹음 파일은 저희 팀 내부에서만 사용하고,
외부에 공유되지 않습니다.

[허락하면] 감사합니다. 지금부터 녹음 시작하겠습니다.
[거절하면] 네, 알겠습니다. 메모로 진행하겠습니다.
```

**팁**:
- 거절당해도 자연스럽게 넘어가기
- 녹음 거절 시 메모에 집중 (눈 마주침 유지)

## 3. 현황 파악 질문 (10분)

### 3-1. 현재 내부 도구 개발 방식

**메인 질문**:
```
요즘 사내에서 내부적으로 사용하는 도구나 시스템은
주로 어떻게 만들고 계신가요?
```

**왜 묻는가?**  
→ 현재 워크플로우 파악. 외주 vs 내부 개발 vs 기성 솔루션 비율.

**경청해야 할 신호**:
- ✅ "개발자가 일일이 코딩해요" → 자동화 니즈 높음
- ✅ "외주 맡기는데 너무 느려요" → 빠른 내재화 니즈
- ⚠️ "Retool 같은 거 써요" → 경쟁자 존재, 불만 파악 필요

**팔로업 질문**:
```
Q1: 최근에 만든 내부 도구 중 하나만 예를 들어주실 수 있을까요?
    - 어떤 용도였나요?
    - 누가 만들었나요? (팀 이름, 인원)
    - 만드는 데 얼마나 걸렸나요?

Q2: 그 과정에서 가장 시간이 오래 걸린 부분은 어디였나요?
    (UI? DB? 배포? 테스트?)

Q3: 완성 후 유지보수는 누가 하나요?
```

**프로브 예시**:
- "예를 들어...?" (구체적 사례 요청)
- "그때 기분이 어떠셨어요?" (감정 파악)

---

### 3-2. 사용 중인 도구들

**메인 질문**:
```
현재 개발팀에서 사용하는 주요 도구나 플랫폼은
어떤 것들이 있나요?
```

**체크리스트** (대화 중 메모):
- [ ] 로우코드/노코드 도구 (Retool, OutSystems, 아웃코드, etc.)
- [ ] 개발 환경 (IDE, Git, CI/CD)
- [ ] 클라우드 vs 온프레미스
- [ ] 데이터베이스 (PostgreSQL, MySQL, Oracle, etc.)
- [ ] 협업 도구 (Jira, Notion, Slack, etc.)
- [ ] AI 도구 (GitHub Copilot, ChatGPT, etc.)

**팔로업 질문**:
```
Q1: 그중에서 가장 만족하는 도구는 무엇인가요?
    - 왜 만족하시나요?

Q2: 반대로 가장 불편한 도구는요?
    - 구체적으로 어떤 점이 불편하신가요?
    - 그럼에도 계속 쓰는 이유는요?

Q3: 새로운 도구를 도입할 때 가장 큰 장벽은 무엇인가요?
    (예산? 보안? 학습 곡선?)
```

---

### 3-3. IT 조직 구조/예산

**메인 질문**:
```
실례가 안 된다면, 현재 IT/개발팀 조직은
어떻게 구성되어 있나요?
```

**파악할 정보**:
- 팀 규모 (개발자 몇 명?)
- 보고 라인 (CTO → 팀장 → 개발자)
- 팀 분리 (프론트/백엔드/인프라/데이터)
- 외주/협력사 의존도

**팔로업 질문**:
```
Q1: 개발자 채용은 잘 되는 편인가요?
    - 어려우시다면 어떤 부분이 어려우신가요?

Q2: 올해 IT 예산은 작년 대비 늘었나요, 줄었나요?
    - 주로 어디에 쓰이나요? (인건비? 라이선스? 인프라?)

Q3: 예산 승인 프로세스는 어떻게 되나요?
    - 팀장님이 결정하시나요, 아니면 더 위로 올라가나요?
```

**주의**:
- 예산 질문은 민감할 수 있음 → "실례가 안 된다면" 전제
- 거절하면 깊게 파지 말고 넘어가기

## 4. 페인포인트 발굴 질문 (15분)

### 4-1. 가장 답답한 순간 구체적 에피소드

**메인 질문 (핵심)**:
```
지난 6개월 동안, 사내 도구 개발이나 운영하면서
"아, 진짜 답답하다"라고 느낀 순간을 
하나만 구체적으로 말씀해 주실 수 있을까요?
```

**왜 중요한가?**  
→ **가장 중요한 질문.** 실제 페인포인트는 추상적 답변이 아니라 구체적 에피소드에 숨어 있음.

**경청해야 할 신호**:
- ✅ "지난주에 이런 일이 있었는데..." → 최근 경험 = 생생한 고통
- ✅ 감정 표현 ("진짜 화났어요", "밤새웠어요") → 진짜 문제
- ⚠️ "일반적으로..." → 추상적 답변, 더 파야 함

**팔로업 질문 (5 Whys 변형)**:
```
Q1: 그때 구체적으로 어떤 상황이었나요?
    - 누가 관련되었나요?
    - 어떤 요청이 들어왔나요?

Q2: 왜 그렇게 답답하셨던 거예요?
    - 기대했던 것과 실제 결과의 차이는?

Q3: 그 문제가 생긴 근본 원인은 무엇이라고 생각하세요?
    (사람? 프로세스? 도구?)

Q4: 그 문제를 해결하려고 시도한 적 있나요?
    - 어떻게 하셨어요?
    - 결과는 어땠나요?

Q5: 만약 그 문제가 완전히 해결된다면,
    어떤 모습일까요? (이상적 시나리오)
```

**침묵 활용**:
- 답변 후 3초 기다리기 → 추가 정보 자연스럽게 나옴
- "음..." 하면 끊지 말고 기다리기

---

### 4-2. 5 Whys 기법 적용 예시

**시나리오**: "사내 재고 확인 앱 만드는 데 3개월 걸렸어요"

```
YOU: 왜 3개월이나 걸렸을까요?
THEM: 요구사항이 계속 바뀌어서요.

YOU: 왜 요구사항이 계속 바뀐 걸까요?
THEM: 현업이 원하는 걸 처음부터 정확히 몰랐어요.

YOU: 왜 처음부터 정확히 파악하지 못한 걸까요?
THEM: 프로토타입 없이 말로만 하다 보니 서로 다르게 이해했어요.

YOU: 왜 프로토타입을 만들지 않으셨어요?
THEM: 프로토타입 만드는 것도 2주씩 걸려서...

YOU: 왜 프로토타입이 2주나 걸릴까요?
THEM: UI 코드 일일이 짜야 하니까요. 디자이너도 없고.

→ **진짜 문제**: 빠른 프로토타이핑 도구 부재
```

**팁**:
- "왜"를 직접적으로 5번 물으면 심문처럼 들림
- 자연스럽게 섞어서: "그 이유가 뭘까요?", "어떤 배경이었어요?"

---

### 4-3. 비용/시간 영향도

**메인 질문**:
```
그 문제 때문에 실제로 얼마나 손해를 보셨나요?
시간이나 비용으로 환산하면 어느 정도일까요?
```

**왜 묻는가?**  
→ 페인의 '강도' 측정. 100만원 문제 vs 1억원 문제는 긴급도가 다름.

**팔로업 질문**:
```
Q1: 그 문제가 한 달에 몇 번 정도 발생하나요?

Q2: 한 번 발생할 때마다 팀원들이 얼마나 시간을 쓰나요?
    - 개발자 시간 = 시급 얼마로 계산?

Q3: 그 시간을 다른 곳에 쓸 수 있다면 뭘 하고 싶으세요?

Q4: 만약 이 문제를 해결하는 솔루션이 있다면,
    얼마까지 투자할 의향이 있으세요?
    (시간적/금전적 투자)
```

**숫자 예시** (대화 중 계산):
```
문제: 사내 앱 하나 만드는 데 2개월 소요
인력: 개발자 2명 풀타임
비용: 2명 × 2개월 × 월급 500만원 = 2,000만원
빈도: 분기당 1회 = 연간 8,000만원

→ "연간 8,000만원 문제네요"라고 정리
```

## 5. AI/보안 관련 질문 (10분)

### 5-1. AI 도구 사용 현황

**메인 질문**:
```
요즘 개발팀에서 ChatGPT나 GitHub Copilot 같은
AI 도구를 사용하고 계신가요?
```

**파악할 정보**:
- 공식 허용 여부
- 실제 사용 여부 (공식 vs 비공식)
- 사용 도구 종류
- 만족도/불만

**팔로업 질문**:
```
Q1: 회사 차원에서 공식적으로 허용하나요, 금지하나요?
    - 허용한다면 어떤 가이드라인이 있나요?
    - 금지한다면 이유가 뭔가요? (보안? 비용?)

Q2: [비공식 사용 언급 시] 회사 몰래 쓰는 분들도 계신가요?
    - 어떤 부분에서 주로 쓰나요?
    - 들키면 어떻게 되나요?

Q3: AI가 생성한 코드를 그대로 쓰시나요, 아니면 검토하나요?
    - 신뢰도는 어느 정도인가요? (0-10점)

Q4: AI 도구 때문에 생산성이 실제로 올라갔다고 느끼세요?
    - 구체적으로 어떤 부분에서요?
```

**주의**:
- 민감한 주제 → "공식적으로는"과 "실제로는" 구분해서 물어보기
- 비공식 사용 언급 시 비밀 보장 강조

---

### 5-2. 망분리 환경 여부

**메인 질문**:
```
혹시 회사 내부망이 인터넷과 분리되어 있나요?
(망분리 환경)
```

**왜 중요한가?**  
→ 온프레미스 필수 여부 결정적 요인. 망분리면 클라우드 솔루션 사용 불가.

**팔로업 질문**:
```
Q1: [망분리 O] 개발할 때 어떻게 하세요?
    - 인터넷 필요할 때는요? (USB? 별도 PC?)
    - npm 패키지 설치는 어떻게 하나요?

Q2: [망분리 X] 그럼 클라우드 서비스 사용은 자유로운가요?
    - AWS, GCP 같은 거 쓰시나요?

Q3: 외부 API 호출은 가능한가요?
    - OpenAI API 같은 걸 내부에서 호출할 수 있나요?
    - 안 된다면 대안은요?

Q4: 망분리 때문에 불편한 점이 있으세요?
    - 가장 큰 제약은 뭔가요?
```

**시그널**:
- ✅ "완전 폐쇄망이에요" → 온프레미스 100% 필수
- ⚠️ "부분 망분리" → 하이브리드 가능성
- ❌ "망분리 없어요" → 클라우드도 옵션

Q: "배포는 어떻게 하시나요?"
   [답변 듣기]
Q: "자동화 도구는요?"
```
→ 한 번에 하나씩.

---

### 9-5. 이분법 질문

❌ **나쁜 예**:
```
Q: "클라우드 쓰세요, 온프레미스 쓰세요?"
```
→ **문제점**: 하이브리드 같은 중간 답변 놓침.

✅ **좋은 예**:
```
Q: "서버 인프라는 어떻게 구성되어 있나요?"
```
→ 열린 질문으로 모든 가능성 포착.

---

### 9-6. 전문 용어 과다 사용

❌ **나쁜 예**:
```
Q: "Orchestration layer에서 
    event-driven architecture 기반으로
    microservices 간 IPC 어떻게 하시나요?"
```
→ **문제점**: 고객이 이해 못 하면 대화 단절.

✅ **좋은 예**:
```
Q: "여러 서비스가 서로 데이터 주고받을 때
    어떻게 하시나요?"
```
→ 쉬운 말로 풀어서.

## 12. 고급 인터뷰 기법

### 12-1. 침묵의 기술

**방법**:
1. 질문 후 최소 3초 기다리기
2. 고객이 "음..." 하면 끊지 말고 기다리기
3. 답변 끝나도 1-2초 더 기다리기 → 추가 정보 나옴

**왜 효과적인가?**  
→ 사람들은 침묵이 불편해서 더 말하게 됨. 가장 중요한 정보는 "추가로" 나옴.

---

### 12-2. 미러링 (Mirroring)

**방법**:
- 고객이 한 말의 마지막 2-3단어를 반복

**예시**:
```
THEM: "사내 앱 만드는 게 너무 오래 걸려요."
YOU: "오래 걸려요?"
THEM: "네, 보통 3개월은 걸리는데..." [추가 설명 시작]
```

**효과**: 자연스럽게 깊이 파기 가능.

---

### 12-3. 감정 라벨링

**방법**:
- 고객의 감정을 언어로 표현

**예시**:
```
THEM: "또 요구사항 바뀌었다고 해서..."
YOU: "정말 짜증 나셨겠어요."
THEM: "맞아요, 그래서..." [진짜 속마음 공유]
```

**효과**: 공감 형성 → 신뢰 → 솔직한 답변.

---

### 12-4. 하향식 질문 (Downward Spiral)

**구조**:
1. 넓은 질문 → 좁은 질문
2. 일반 → 구체적
3. 최근 → 과거

**예시**:
```
Q1: "사내 도구 개발은 어떻게 하시나요?" (넓음)
Q2: "최근에 만든 도구 중 하나만 예를 들어주실 수 있나요?" (좁음)
Q3: "그 프로젝트 시작할 때 첫 단계는 뭐였나요?" (구체적)
```

---

### 12-5. 가상 시나리오 테스트

**방법**:
- 가정이 아니라 '현실화된 가정'

❌ **나쁜 예**:
```
"만약 완벽한 솔루션이 있다면 쓰시겠어요?"
```

✅ **좋은 예**:
```
"만약 다음 달에 베타 버전이 나온다면,
한 달 동안 무료로 써보시겠어요?
단, 버그가 있을 수 있고, 기능이 부족할 수 있습니다."
```

**효과**: 진짜 구매 의도 vs 립서비스 구분.

---

## 13. 체크리스트: 좋은 인터뷰 vs 나쁜 인터뷰

### 좋은 인터뷰의 신호 ✅

- [ ] 고객이 70% 이상 말함 (나는 30% 이하)
- [ ] 구체적 에피소드 3개 이상 수집
- [ ] 숫자/데이터 확보 (비용, 시간, 빈도 등)
- [ ] "아, 그건 처음 생각해 봤네요" 같은 반응 유도
- [ ] 감정 표현 ("짜증났어요", "기뻤어요") 확인
- [ ] 다음 단계 약속 (레퍼런스, PoC 등)
- [ ] 고객이 질문 던짐 (관심 있다는 증거)
- [ ] 시간 초과 (60분 → 90분) → 몰입한 대화

---

### 나쁜 인터뷰의 신호 ⚠️

- [ ] 내가 70% 이상 말함 (설명, 영업)
- [ ] "일반적으로는..." 같은 추상적 답변만
- [ ] 구체적 숫자 없음
- [ ] "그냥 그래요" "잘 모르겠어요" 반복
- [ ] 감정 없음 (건조한 대화)
- [ ] "나중에 연락 주세요"로 끝
- [ ] 고객 질문 없음 (관심 없음)
- [ ] 정확히 60분에 끝냄 (빨리 끝내고 싶어함)

    현재 사내 도구는 어떻게 개발하시나요?"
   ```

2. **망분리 확인**
   ```
   "망분리 환경이시라고 들었는데,
    그럼 외부 API나 클라우드 서비스는 아예 못 쓰시나요?"
   ```

3. **페인포인트**
   ```
   "그런 환경에서 개발하실 때 가장 답답한 부분이 뭔가요?"
   ```

4. **AI 도구**
   ```
   "ChatGPT 같은 AI 도구도 못 쓰시겠네요?
    대안이 있으신가요?"
   ```

5. **의사결정**
   ```
   "새로운 솔루션 도입하려면
    보안 검토가 필수일 것 같은데,
    어떤 절차를 거치나요?"
   ```

## 15. 마지막 조언

### 인터뷰는 대화지 심문이 아니다

- 질문 가이드는 '대본'이 아니라 '나침반'
- 고객이 흥미로운 방향으로 가면 따라가기
- "대본대로 안 됐다"고 실패한 게 아님

---

### 80/20 법칙

- 20%의 질문이 80%의 인사이트를 만듦
- 핵심 질문: **"가장 답답했던 순간을 구체적으로 말씀해 주세요"**
- 이 질문 하나만 잘 파도 성공적 인터뷰

---

### 실패는 배움이다

- 첫 인터뷰는 어색함 → 정상
- 10번째 인터뷰부터 패턴 보임
- 50번 인터뷰하면 고수

---

### 녹음을 다시 들어라

- 인터뷰 직후 들으면 놓친 부분 발견
- "왜 여기서 더 안 물어봤지?" → 다음 인터뷰 개선
- 자기 목소리 듣기 민망하지만 필수

---

### 인터뷰이를 존중하라

- 시간 지켜주기 (60분 → 70분 MAX)
- 비밀 보장 (녹음 파일 유출 금지)
- 약속 지키기 (답례품, 피드백 공유 등)
- 진심 어린 감사 표현
